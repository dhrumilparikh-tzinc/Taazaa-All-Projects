# Medical RAG API

A REST API that answers medical questions using **Retrieval-Augmented Generation (RAG)**. Users submit a natural-language question; the system finds the most relevant passages from a 16,000-entry medical knowledge base and generates a grounded answer via Google Gemini.

This was the **first RAG project** of the internship — built to understand the core retrieve-then-generate loop before scaling to multimodal systems.

---

## How It Works

```
User question
      │
      ▼
Embed question (SentenceTransformer all-MiniLM-L6-v2, 384-dim)
      │
      ▼
FAISS similarity search over 61,886 medical chunks
      │
      ▼
Top-k chunks assembled into prompt context
      │
      ▼
Gemini LLM generates grounded answer
      │
      ▼
Answer + source chunks returned (+ logged to PostgreSQL)
```

### Dataset — MedQuAD
- **Source:** Medical Question Answering Dataset (NIH / NLM)
- **Documents:** 16,407 medical Q&A pairs
- **Chunks:** 61,886 (500-char chunks, 100-char overlap)
- **Topics:** Endocrinology, Cardiology, Neurology, Oncology, and more

## Tech Stack

| Layer | Technology |
|---|---|
| API Framework | FastAPI + Uvicorn |
| Embeddings | `sentence-transformers/all-MiniLM-L6-v2` (local) |
| Vector Search | FAISS (L2 index, CPU) |
| LLM | Google Gemini 2.5 Flash |
| Database | PostgreSQL (SQLAlchemy ORM) |
| Data | MedQuAD CSV (16k medical Q&As) |

## Setup

### Prerequisites
- Python 3.10+
- PostgreSQL running locally
- Google Gemini API key ([get one here](https://aistudio.google.com/))

### 1 — Install dependencies

```bash
cd basic-rag-main
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 2 — Configure environment

```bash
cp .env.example .env
```

Edit `.env`:
```env
GEMINI_API_KEY=your_gemini_api_key_here
GEMINI_MODEL=gemini-2.5-flash
DATABASE_URL=postgresql+psycopg2://postgres:yourpassword@localhost:5432/basic_rag
```

### 3 — Initialize the database

```bash
createdb basic_rag
python -m scripts.init_db
```

### 4 — Build the FAISS index

```bash
python -m scripts.build_index
```

Loads `data/medquad.csv`, chunks all documents, embeds them, and saves the FAISS index to `faiss_index/`. Takes ~2 minutes on first run. The pre-built index is included so this step is optional.

### 5 — Start the API

```bash
uvicorn api:app --host 0.0.0.0 --port 8000 --reload
```

API live at `http://localhost:8000` — interactive docs at `http://localhost:8000/docs`.

## API Endpoints

### `POST /ask`
Ask a medical question. Returns the answer plus the retrieved source chunks.

**Request:**
```json
{
  "question": "What are the symptoms of Type 2 diabetes?",
  "k": 5
}
```

**Response:**
```json
{
  "status": "success",
  "question": "What are the symptoms of Type 2 diabetes?",
  "answer": "Type 2 diabetes symptoms include increased thirst, frequent urination, fatigue...",
  "chunks": [
    {
      "text": "Type 2 diabetes is characterized by...",
      "metadata": { "source": "NIHSeniorHealth", "focus_area": "Diabetes" },
      "distance": 0.312
    }
  ],
  "num_chunks": 5
}
```

### `POST /ask/simple`
Same as `/ask` but returns only question and answer — no chunk metadata.

### `GET /health`
Returns `{"status": "healthy"}`.

### `GET /`
Returns API version info.

## Key Design Decisions

- **Local embeddings** — `all-MiniLM-L6-v2` runs on CPU with zero API cost for indexing
- **FAISS over a hosted vector DB** — Simple, fast, no infrastructure needed for a single-dataset project
- **Graceful logging** — If PostgreSQL is unavailable, the API still answers (logging is best-effort)
- **Pre-built index** — The FAISS index is committed so new contributors can run the API immediately without rebuilding
